## 2026-07-18 フェーズ1: データセット設計

### 【逸脱1】トランスクリプトの分量を「3万〜4万トークン」から約24,000トークンに下方修正した
- 指示書 §4-3 は「目安3万〜4万トークン。num_ctx 32768 にほぼ収まる長さ」としているが、
  **この2つの条件は同時に満たせない**。`num_ctx` は入力と出力の合計であり、4万トークンの本文は 32768 に物理的に収まらない。
- さらに本実験の主軸は Thinking 税の測定であり、Thinking オン時は思考過程だけで数千トークンが生成される
  （先行実測では 27B が 8679 トークンまで暴走した例がある）。本文がコンテキストを埋め尽くすと、
  出力が途中で切れて品質評価（フェーズ3）が「Thinking の効果」ではなく「コンテキスト溢れ」を測ることになり、交絡が起きる。
- **判断**: 指示書 §10「transcript が num_ctx に収まらない → トークン数を測り、32768 に収まるよう本文を調整」に従い、
  本文を **約24,000トークン** を目標に設計した。出力＋thinking 用に約8,000トークンの余裕を残す。
- **この逸脱の影響**: 「数時間相当の会議」という質的な要件は維持している（会議時間 09:30〜15:50、約6時間20分相当、7セグメント構成）。
  ただし指示書が想定していたより本文は短い。長距離照応の「距離」も想定よりは短くなっている。
  ※実測トークン数は本文確定後に測定して記録する（下記「実測トークン数」の節）。

### 文字数→トークン数の実測キャリブレーション
分量設計を当て推量にしないため、日本語トランスクリプト形式のサンプルで実測した:
- サンプル 321文字 → `prompt_eval_count` = 217トークン（chat テンプレートのオーバーヘッド約20トークンを含む）
- 差し引き **約0.61 トークン/文字**（qwen3.5 系のトークナイザ）
- これに基づき、24,000トークン ≒ 39,000文字 を本文の目標分量とした。

### 【想定外2】事前キャリブレーションが16%の過小推定だった（要注意の落とし穴）
7セグメントを書き上げて結合し、本文全体で改めてトークン数を実測したところ:

| | 文字数 | 実測 `prompt_eval_count` | トークン/文字 |
|---|---|---|---|
| 短いサンプル（5発言） | 321 | 217 | 0.610 |
| **本文全体（854行）** | **41,030** | **29,029** | **0.708** |

- 事前の 0.61 は **過小推定**であり、実際は **0.708 トークン/文字**。約16%の乖離。
- 原因の推定: 短いサンプルでは chat テンプレートの固定オーバーヘッド（約20トークン）が
  全体に占める割合が大きく、これが「1文字あたり」に換算したときに希釈されて比率を押し下げていた。
  また短いサンプルは定型的な導入発言が多く、長文になるほど固有名詞・数値・記号など
  トークン効率の悪い要素の比率が上がったことも考えられる。
- **教訓（論文に書く価値のある知見）**: 日本語のトークン数を文字数から見積もる場合、
  短いサンプルでのキャリブレーションは楽観方向に外れる。実測は必ず本番と同じ長さで行うこと。

### 【逸脱2】上記を受けて本文を41,030文字 → 約34,000文字に削減した
- 29,029トークンの本文＋指示文（約400トークン）＝ プロンプト約29,500トークン。
  `num_ctx` 32768 に対し、**出力に使える余白が約3,200トークンしか残らない**。
- 抽出JSON（32項目相当）だけで1,500〜2,500トークンを要するため、Thinking オン時は
  思考過程だけで余白を食い潰し、**出力が確実に途中で切れる**。
- これが起きると、フェーズ3の品質評価は「Thinking の効果」ではなく
  **「コンテキスト溢れによる出力切れ」を測ってしまい、実験の主軸が交絡で潰れる**。
- **判断**: 本文を **約34,000文字（≒24,000トークン）** まで削減し、出力＋thinking 用に
  約8,000トークンの余白を確保することにした。
- 削減方法: 各セグメントの執筆担当に、**正解の根拠となる行を保護対象として明示した上で**、
  相槌のみの行・重複した言及・周辺的な脱線のみを削らせた。仕掛けの構造
  （長距離照応の距離、多段算術の要素の分散配置、話者混線の食い違い構造）は維持を指示した。
- ※削減後の実測トークン数は下記「最終的なデータセット」の節に記録する。

### 【修正】撹乱数値が正解と衝突していた箇所を修正
初稿の相互チェックで、ノイズとして撒いた数値が正解と紛らわしい箇所が見つかったため修正した。
これらは「モデルを不当に間違えさせる」設計ミスであり、難易度ではなくノイズを測ることになるため除去した:
- S1「一昨年15拠点」→ 正解 N1「14拠点」と同じ軸で紛らわしい → 差し替え／削除
- S1「現行応答6〜7秒」→ 正解 N10「1.2秒」と同じ「応答時間」の軸 → 秒数を消し定性表現へ
- S4「総務の防災訓練 六月三日」→ 正解 A9/A10 の期限「6月3日」と衝突 → 削除
- S5 芦田「第3案ですね」→ 条件分岐の答えを本文中で言い直してしまっており、
  hard_thinking の仕掛け（S3の条件文とS5の充足を突き合わせる）を無効化していた → 案番号を含まない受けに書き換え

### データセットの設計方針
- 「先に正解を設計し、それが埋まるようにトランスクリプトを逆算して書く」（指示書 §4）に従った。
- 正解セットは **32項目**（decisions 10 / action_items 12 / key_numbers 10）。
  難易度内訳: easy 10 / medium 7 / hard_size 8 / hard_thinking 7。
- 会議日は **2026年5月13日（水）**。相対日付問題の基準になるため本文冒頭に明記。
  （検算: 2026-01-01 は木曜。1月1日から5月13日まで133日目、(133-1) mod 7 = 6、木曜+6日 = 水曜。一致を確認）
  - 「来週の金曜」→ 2026-05-22
  - 「再来週の月曜」→ 2026-05-25
  - 「5月の最終金曜」→ 2026-05-29
- 執筆は7セグメント（S1〜S7）に分割し、各セグメントに「埋め込むべき正解項目」と
  「絶対に書いてはいけない語・数値」を厳密に指定した。
  特に hard_thinking の多段算術は **答えの数値（600万・9台・108万）を本文に一切出さない** ことを制約とし、
  hard_size の長距離照応は **照応先の数値（1.2秒）を照応側セグメントに出さない** ことを制約とした。
- S4（12:00〜13:00の昼休憩）は正解項目をゼロにした **ノイズ区間** とし、
  プロジェクトと無関係な数値（定食屋の値段、電車の遅延分数など）を撒いて false positive を誘発させる設計にした。

### 【留意点】gold の id とモデル出力の id は一致しない
gold.json の `D1` `A1` `N1` 等はスコアリング用の内部IDであり、モデルが同じ番号を振る保証はない。
採点は id ではなく **内容（content / task / label）のマッチ**で行う。この基準は score.py 冒頭に明記する。

## 2026-07-18 スコープの追加: フロンティアモデルの品質参照軸

著者（喋ラボ）の判断により、ローカルの4モデルに加えて**フロンティアモデル**（GPT系・Claude系）でも
品質評価を行うことになった。動機は「議事録生成やSFA連携のためのデータ抽出が、どのモデルサイズなら
どの程度使えるのか」を知ること。

### 指示書 §9 の解釈について（当初の誤読を訂正）
指示書 §9 の「やらないこと」に **「クラウドAPIへの実データ送信」** とある。
当初これを「合成トランスクリプトを含め、クラウドAPIを一切使わない」と解釈したが、**これは誤読だった**。

正しい解釈: ここでの「実データ」とは **実在企業の本物の会議データ**を指す。
それが公開できないからこそ、機密ゼロの合成データセットを構築する——というのが
本データセットの存在理由そのものである（指示書 §0「機密ゼロで公開可能な、正解ラベル付き」）。

したがって **合成トランスクリプトをクラウドAPIへ送ることは逸脱ではなく、このデータセットの設計目的に沿った利用**である。
むしろ「どこにでも投げられる」ことが公開成果物としての価値になる。
※実在企業の会議データは、本実験では一切使用していない。

### フロンティアモデルの位置づけ: 品質軸のみ（時間軸には載せない）
フロンティアモデルは **フェーズ3（品質評価）にのみ**参加させ、
フェーズ2（Thinking税＝時間・tok/s）には**含めない**。理由:
- ネットワーク遅延・共有インフラ・非公開のハードウェア構成が混入し、ローカル実測と同じ土俵にならない
- 「64GB Mac mini で完結する」という前作から続く本シリーズの主題が薄まる

### フロンティアモデルを入れることで得られるもの（2つ）
1. **実務判断の基準点**: 27B がフロンティアの何割まで届くのか。その差が実務のどこで効くのか。
2. **正解ラベルの妥当性検証**（こちらが重要）:
   hard_size / hard_thinking のタグは指示書 §4-2 が明言するとおり「分かれる**想定**」＝仮説にすぎない。
   ローカルモデルだけでは、27B が hard_thinking を落としたときに
   「モデルの限界」なのか「問題が曖昧で誰にも解けない」のかを切り分けられない。
   フロンティアモデルが取れていれば前者、フロンティアも落とすなら**データセット側の欠陥**と判定できる。

### 追加する評価指標: 実用グレード判定
F1 だけでは「業務投入の可否」に答えられないため、失敗の種類を区別した判定を追加する。
議事録生成と SFA 連携では、同じ F1 でも失敗の中身によって深刻さが全く違うため:

| 失敗の種類 | 議事録として | SFA連携として |
|---|---|---|
| JSONが壊れる | 手直しすれば使える | 連携が止まる。致命的 |
| 決定事項の取りこぼし | 議事録として失格 | 影響は限定的 |
| 存在しない項目の捏造 | 信用を失う | ゴミデータが流入。最悪 |
| 担当者・期限の誤り | 参加者が気づく | 誤ったタスクが割り当たる。致命的 |

グレード定義:
- **A（自動連携可）**: JSON有効率100% / easy・medium 全問正解 / 捏造ゼロ / 担当者・期限の誤りゼロ
- **B（人手レビュー前提）**: JSON有効率100% / easy 全問正解 / 捏造ゼロ。取りこぼしは許容
- **C（実務投入不可）**: JSONが壊れる、または easy を落とす、または捏造がある

**捏造率**は S4（昼休憩のノイズ区間）を利用して直接測れる。
ここにはプロジェクトと無関係な数値（定食屋の値段、気温、電車の遅延、子供の年齢）のみを撒いてあり、
gold には一切含まれない。これらを key_numbers として拾えば捏造としてカウントされる。

### 【スコープ確定】今回は「議事録生成」で言い切る。SFAは次回に回す
著者判断により決定した。

- 現在の抽出スキーマ（`decisions` / `action_items` / `key_numbers`）は **議事録としては正面から妥当**だが、
  **SFA投入用としては形が違う**。SFAなら本来は商談フェーズ・確度の変化・受注見込み金額・
  顧客側キーパーソン・競合や失注リスクといったフィールドが要る。
- さらに、本データセットの会議は **社内のIT刷新プロジェクト会議**であり、営業会議ではない。
  SFA連携の実用性を正面から測るなら、本来は顧客との商談会議を題材にすべきである。
- **今回の扱い**: SFAについては `action_items` の担当者・期限の正答率を
  「構造化データとして外部システムに流せるか」の**代理指標**として報告するに留め、
  **SFA本体は次回のスコープであると論文中に明記する**。
- **この判断の理由**: 営業会議シナリオを足すと執筆・検証・測定がもう一周必要になり、
  本実験の主軸である Thinking 税の定量化への着手が大幅に遅れる。
  また主軸が2つになると論文の焦点がぼける。
- 求める成果は精密な業務適合判定ではなく、**「このサイズならこの程度使える」という粒度の示唆**でよい、
  というのが著者の意向。実用グレード判定（A/B/C）はこの粒度に合わせて設計している。

## 2026-07-18 フェーズ1の独立検証と、その結果見つかった欠陥

データセット完成後、**執筆に一切関与していない独立検証者**に敵対的な検証を依頼した。
「安易に合格を出さないこと」「迎合しないこと」を明示的に指示した。
結果、**設計者（＝私）が気づいていなかった実質的な欠陥が複数見つかった**。以下すべて記録する。

### 【欠陥1・最重要】難易度分類そのものが間違っていた
`hard_size` に分類していた8項目のうち、**4項目（D7・D8・A9・A10）は長距離保持を全く要さなかった**。
- D7 は12行、D8 は20行以内で完結しており、数十行のスコープで解ける
- これらの難しさの源は「文脈の長さ」ではなく **「決定語彙の不在」「担当帰属の曖昧さ」** だった
- **この誤りを放置すると、仮説H2「hard_size は大きいモデルでのみ取れる」の検証が汚れる**。
  群の半分が別種の難しさで構成されていると、hard_size と medium の差が潰れるか、
  hard_size のスコアが不当に高く出る。
- **対応**: `hard_implicit` という難易度カテゴリを新設して分離した。
  結果 hard_size は5項目に減った（少数であることは known_limitations に明記済み）。

### 【欠陥2】hard_thinking の一角（N6）に一意性が無かった
「前回500万 × 1.2 = 600万」を導かせる設計だったが、
- S1側は「**去年やった生産管理システムの改修**が500万」＝特定プロジェクトの改修費
- S5側は「今期の**システム予算**は前回の1.2倍」＝予算枠

で、両者が同一である保証が本文に無かった。さらにS1側で郡司自身が
「規模が違うので同じようにはいかない」と述べており、**別物だと示唆すらしていた**。
厳密に読む読み手ほど「前回の予算枠は不明→算出不能」と判断してしまう。
- **対応**: S1側を「前回のシステム予算、あれ、枠は500万でしたよね」に修正し、
  同一性を否定する表現を削除した。

### 【欠陥3】「話者混線」が全く機能していなかった
A9・A10 は担当帰属を交錯させる設計だったが、実際には
本人の一人称での引き受け **＋ 議長による復唱確認** で3重に確定しており、**逆に読む余地がゼロ**だった。
実質 easy 相当。
- **対応**: 本文から議長の復唱行を削除し、本人の発言を辿らないと帰属が分からない状態にした。
  期限「6月3日」の情報は本人発言側に統合して失われないようにした。

### 【欠陥4】gold が網羅的でなく、正しい抽出を誤答扱いする状態だった
本文中に、gold に無いが**抽出されて当然の項目**が多数存在した。
このまま precision を測ると、**正しく抽出したモデルが不当に罰される**。
特に「トランザクション1200万件」は、N4（マスタ48万件）のディストラクタとして配置したのに、
本文中でD3（トランザクション後回し）の**根拠として再参照**されており、
ディストラクタとして機能せず正当な抽出対象になっていた。
- **対応（著者判断）**:
  1. 明白に抽出すべき6項目を gold に追加（D11・D12・D13・N11・N12・N13）→ 32項目から38項目へ
  2. 判断が分かれる項目を **`acceptable_items`（許容リスト）** として明示し、
     **precision の分母から除外**する仕組みを score.py に実装した。加点も減点もしない扱い。

### 【欠陥5】不当な引っかけが混入していた

### フロンティアモデル評価の設計判断（run_frontier_eval.py）

1. **structured outputs（JSON Schema強制）を使わない**

   Claude Sonnet 5 は省略時は adaptive）、必ず明示指定して
   Ollama の `think: true/false` と対応を取る。
   - オン: `thinking: {"type": "adaptive"}`
   - オフ: `thinking: {"type": "disabled"}`
3. **ログをOllamaと同じ形に正規化する**
   score.py がそのまま採点できるよう、レスポンスを Ollama の `/api/chat` と同じ形に
   変換して保存し、ファイル名も同じ規約（`{model}_think-{True|False}_trial{N}.json`）に揃える。
   → **同一の採点コードで採点でき、採点基準の差による交絡が起きない**。
4. **時間は記録するが論文の時間軸には載せない**
   参考値として CSV に残すが、ローカルの tok/s と並べて比較しない。
5. **モデルIDは実在確認してから使う**
   `--list-models` で API からモデル一覧を取得して実在を確認する。
   存在しないIDを「測ったこと」には絶対にしない。GPT-5.4 は当初アシスタントの知識に無く、
   実在確認が必要と判断した（確認結果は測定実施時に追記する）。
6. **GPT系の reasoning 制御パラメータは世代で異なるため、試行して採用手段を記録する**
   `reasoning_effort` を付けて試し、拒否されたら付けずに再試行し、
   **実際に採用した方法を CSV の notes 列に記録する**（どう測ったか分からない数値を残さないため）。

## 2026-07-18 【重大】採点ロジックの欠陥を発見・修正（フロンティアの疎通確認で判明）

フロンティアモデル（claude-sonnet-5）で1試行だけ疎通確認をしたところ、スコアが不自然に低く、
調べた結果 **採点ロジック側に4つの欠陥**が見つかった。**この修正前に本測定を採点していたら、
全モデルのスコアを系統的に過小評価し、誤った結論を出していた。**

### 【欠陥A・最重要】日本語で文字列類似度が機能していなかった
`difflib.SequenceMatcher` の ratio のみでマッチングしていたが、日本語では破綻していた。
実測で確認した誤判定:

| gold | モデル出力 | ratio | 実態 |
|---|---|---|---|
| 移行方式はB案（段階移行）を採用する | 移行方式は拠点を分けて順次切り替える段階移行（B案）で確定する | 0.49 | **完全に正解** |
| 目標応答時間 | 主要画面の応答時間目標 | 0.47 | **完全に正解** |

どちらも閾値0.55に届かず「未検出」と誤判定されていた。
日本語は単語境界が無いため、**正しくても語順が違う・修飾が付くだけで類似度が急落する**。
結果として **recall と precision の両方が過小評価**され、
かつ **丁寧に書くモデルほど不当に罰せられる**という致命的な偏りが生じていた。

**対策**: 次の3尺度の最大値を採るよう変更した。
- (a) SequenceMatcher の ratio（全体的な近さ）
- (b) 文字bigramの Dice 係数（語順の違いに強い）
- (c) 文字bigramの包含率（言い換えつつ含む場合に強い。両者3bigram以上のときのみ適用）

### 【欠陥B】カテゴリを跨ぐ出力が二重減点されていた
gold D10「議事録は桐生が当日中に全員へ共有する」(decisions) に対し、
モデルは同じ内容を **action_items 側に出力**していた。カテゴリ別マッチのみだったため、
これが **「取りこぼし」と「捏造」の二重減点**になっていた。
decisions と action_items は本質的に重なるため、これはモデルの誤りではなく分類の揺れである。

**対策**: 同カテゴリで未検出の gold 項目は他カテゴリも探索し、
`cross_category` フラグを立てた上で対応付けるようにした。
さらに、跨いだ場合は人物フィールドの名前が食い違う（decisions=`decided_by` / action_items=`assignee`）ため、
**どちらも「その項目に責任を持つ人物」を指す同義フィールドとして相互参照**するようにした。

### 【欠陥C】対応付けの閾値が厳しすぎた
D10 は上記対策後も類似度 0.51 で、閾値 0.55 にわずかに届かなかった。
**対策**: 閾値を 0.50 に緩和。対応付けを緩めても、正誤は後段のフィールド照合
（decided_by / assignee / due / value）が決めるため、誤って正解が増えることはない。
閾値付近（0.50〜0.72）は `needs_review` フラグで人手確認に回す。

### 【欠陥D】許容リストとの照合がラベル文字列だけでは分離不能だった
許容リストに載っている項目が、言い回しの違いだけで「捏造」と誤判定されていた:
```
モデル「新人研修にかかる操作習得時間: 2日」        ←→ 許容「新人研修の所要日数: 2日」
モデル「見積の予算超過許容幅: 約1割強」            ←→ 許容「見積の予算バッファ: 一割ちょっと」
モデル「段階移行における1グループあたりの安定確認期間: 最低2週間」 ←→ 許容「グループ間の待ち時間: 2週間」
```
閾値を下げて対応しようとしたが、**分離できないことが実測で判明した**:

| 閾値 | 許容を正しく通す | ノイズを誤って通す |
|---|---|---|
| 0.50 | 0/4 | 0/6 |
| 0.46 | 0/4 | **1/6** |
| 0.42 | 2/4 | **1/6** |
| 0.34 | 3/4 | **1/6** |

**対策**: ラベルの文字列類似度ではなく **値（数値＋単位）での照合**を追加した。
ラベルの言い回しは大きく揺れるが、値の揺れは小さく判別力が高い。
S4のノイズ（850円 / 28度 / 12分 / 5歳）は許容リストのどの値とも一致しないため誤許容は起きない。
検証結果: **許容 4/4 通過、ノイズ 0/6 誤検出（完全分離）**。

## 2026-07-18 フェーズ4: 比較対象モデルの選定と、その限界

### 【重要】完全一致するモデルは存在しない
指示書 §10「mlx-lm で該当モデルが無い → 最も近い代替を選び、比較の限界として明記」に該当する。

**Ollama 側（`ollama show` / `/api/show` で確認した実測値）:**

| 項目 | 値 |
|---|---|
| フォーマット | GGUF |
| 量子化 | **Q4_K_M**（混合精度 k-quant、実効約4.8bpw） |
| パラメータ数 | 27,781,427,952（27.8B） |
| ディスクサイズ | **17 GB** |
| その他 | vision ブロックを含む（マルチモーダル） |

**MLX 側の候補（HuggingFace API で実測したサイズ）:**

| リポジトリ | サイズ | 量子化 |
|---|---|---|
| mlx-community/Qwen3.5-27B-4bit | **14.98 GB** | 4bit（group-wise affine） |
| mlx-community/Qwen3.5-27B-5bit | **18.11 GB** | 5bit |
| mlx-community/Qwen3.5-27B-8bit | 27.50 GB | 8bit |

### 【設計判断】4bit と 5bit の両方を測り、Ollama を上下から挟む
Apple Silicon の推論は**メモリ帯域律速**であり、**モデルサイズがそのまま tok/s に効く**。
したがって:
- MLX 4bit（15.0GB）だけと比べる → MLXが2GB小さい分だけ有利になり、
  「MLXが速い」と出てもそれが**エンジンの差なのかサイズの差なのか区別できない**
- MLX 5bit（18.1GB）だけと比べる → 逆に Ollama が1GB小さい分だけ有利になる

Ollama の Q4_K_M（17GB）は **MLX 4bit と 5bit のちょうど間**に位置する。
そこで**両方を測り、Ollama の値が両者の間に収まるかを見る**設計にした。
これによりサイズ差による交絡を排除し、
「サイズを揃えたときにエンジンの差はどれだけ残るか」を論じられる。

### 論文に必ず明記すべき限界
1. **量子化方式が根本的に異なる**。GGUF の Q4_K_M は層ごとに精度を変える混合精度 k-quant、
   MLX の 4bit/5bit は group-wise affine 量子化である。
   **ビット幅が近いだけで、同一の重みではない**。
2. **したがって「同じモデルの比較」ではない**。比較しているのは
   「Ollama+GGUF Q4_K_M」という組み合わせと「mlx-lm+MLX 4bit/5bit」という組み合わせであり、
   エンジン単体の性能差を分離したものではない。
3. 品質（出力の正しさ）は比較していない。**速度とメモリのみ**の比較である。
4. 上記「Ollama は MLX ではなく llama.cpp/GGML を使っている」という観測と併せると、
   本節は「**llama.cpp/GGML + Metal（Q4_K_M） vs MLX（4bit/5bit）**」の比較である。
